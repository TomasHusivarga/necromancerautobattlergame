Solo-dev 2D strategy/autobattler concept

Build the monster, then watch the battlefield prove it.

The player runs a necromancer lab, stitches zombies from modular corpse parts, and sends those designs into automatic lane battles. The game is about engineering bodies, learning from fights, and improving designs between runs.

3 mo target for first vertical slice
1 lane initial battle scope
15 first production body parts
243 base combinations from 3 parts per slot

Core hook

Corpse-engineering: heads, torsos, front arms, back arms, legs, and mutations create zombie roles. The modular rig is the production backbone.

Hard boundary

No multiplayer, campaign sprawl, heavy crafting, or unique full-body sprites until the solo loop proves it is fun.

Core Loop

The whole game should stay readable as a repeatable lab-to-battle cycle.

Choose corpse parts

Pick from owned heads, torsos, arms, legs, and mutations.

Assemble zombie

Preview the body, compare stats, and save the design.

Start lane battle

Zombies spawn, march forward, and fight automatically.

Read the result

See what failed: speed, armor, range, plague, sustain, or damage.

Improve design

Unlock parts, upgrade pieces, mutate bodies, and try again.

Asset Pipeline

The whole concept depends on reusable modular parts, not unique sprites for every combination.

Correct direction

Paper-doll zombie rig: full-canvas transparent PNGs layered in a strict order. Same camera, scale, ground line, and anchor points.

Main production risk

AI can make cool concepts, but generated parts may not attach cleanly. The risky part is geometry consistency, not visual style.

First rig

512x512 canvas, right-facing side view, body center x=256, ground line y=430, layer order: back arm, legs, torso, head, front arm, effects.

QA rule

A part is not accepted until it composites cleanly with parts from other families. It cannot only work with its original matching set.

Later 1v1 Expansion

Only consider multiplayer if early access proves the solo game has strong retention.

1v1 pitch

Two players bring zombie designs, spawn from opposite sides, and automatic lane combat determines whose corpse-engineering is stronger.

Best first mode

Budget Duel: both players build within the same point budget so PvP does not become unfair account progression.

Do not build now

No accounts, matchmaking, servers, rankings, anti-cheat, or real-time sync in the first version.

Future-proof lightly

Keep battles deterministic and part data clean. That helps replays and multiplayer later without turning MVP into a networking project.

What Not To Build First

These ideas can be good later, but they are dangerous before the core loop is proven.

No full campaign

Three short battles are enough for the first playable slice.

No huge crafting

Use simple upgrades and one mutation slot before recipes and experiments.

No unique body sprites

The production plan must survive hundreds of combinations.

No multi-lane start

One lane keeps AI, UI, and balance manageable.

No PvP start

Multiplayer is a later expansion, not the first product.

No invisible complexity

Effects, failure, and counters must be visible to the player.
